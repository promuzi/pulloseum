> **범위:** 이 문서는 *방향과 2기둥 아키텍처, 그리고 지금부터 지킬 규율*을 못박는다. 엔드포인트·DB 스키마 등 상세 스펙은 **실제 서버 작업 직전에** 따로 쓴다(지금 다 쓰면 게임이 계속 바뀌어 썩는다).

## 0. 한 줄 요약

치트(특히 "재화·식물 만렙을 조작해 만든 뒤 PvP 진입")를 **출시 때부터 원천봉쇄**하기 위해 **풀 서버 권위**로 간다: 진행(경험치·재화·식물)을 서버만 발급하고, 전투 결과를 서버가 재검증한다. 게임 로직이 전부 JS라 **같은 결정론적 시뮬 코어를 브라우저와 서버(Node)에서 공유**하는 게 핵심 열쇠. 온라인 필수. 출시가 멀어 시간은 충분하나, **지금부터 결정론 규율을 지켜 미래 추출 비용을 줄인다**(§5, CLAUDE.md 하드 지침).

## 1. 왜 풀 서버 권위인가 (blob 안을 폐기한 이유)

브레인스토밍 합의:
- **목표(궁극):** ① 수익화(인앱결제) ② 크로스기기 세이브 ③ PvP/랭킹/시즌.
- **백엔드:** Firebase로 시작(Auth+Firestore), 전투 재검증엔 **Cloud Functions/작은 Node 서버**.
- **치트 방지가 1순위 제약.** 핵심 위협 = "재화·레벨을 클라에서 조작 → 만렙 식물로 PvP 진입."

**"나중에 PvP 붙이며 막자"가 안 되는 이유:** 클라이언트 권위로 출시하면 PvP 출시 전에 이미 치트로 만렙을 만든 계정이 존재 → PvP 첫날 그대로 진입 → 소급 적발은 지옥. **처음부터 서버가 진실을 쥐는 것만이 깔끔한 해법.**

**클라 쪽 트릭은 전부 무의미:** 암호화·난독화·"와이파이 강제"로는 못 막는다(폰은 공격자 손 안). 유일한 진짜 방어 = **서버가 진행을 발급·검증**. "온라인 필수"는 그 결과로 자연히 따라온다(사용자가 직관적으로 원했던 "와이파이 필수"의 진짜 정체).

**단계 선택(확정):** "서버 진행 발급 + **전투까지 서버 재검증**"(최상위). 진행 위조와 전투 결과 위조를 모두 차단.

## 2. 아키텍처 — 기둥 2개

### 기둥 1 — 서버 권위 계정 (진행을 서버만 발급)
```
경험치·재화·식물·아이템 = 서버가 보관·지급. 폰은 캐시(view)일 뿐.
모든 획득(탐사 보상·양육 열매·전투 보상·구매)은 서버를 거쳐 확정.
→ 폰은 상태를 '만들어낼' 수 없다. "치트로 만렙" 자체가 성립 불가.
```
- 이 기둥 하나만으로 사용자의 핵심 걱정("조작 만렙 → PvP")은 끝난다.
- 크로스기기 세이브는 **공짜로 따라옴**(계정이 서버에 사니까 = 어느 기기서 켜도 서버 상태 로드).

### 기둥 2 — 서버 전투 재검증 (결정론적 공유 시뮬)
```
전투엔진을 결정론적으로 만든다: (내 로스터, 상대 로스터, 시드, 입력열) → 항상 같은 결과.
폰: 즉시 연출용으로 시뮬 실행(체감 빠름).
폰 → 서버: 입력열 + 주장 결과 제출.
서버: 같은 결정론 엔진을 서버가 아는 로스터로 재실행 → 결과 일치+승리면 보상 지급, 불일치면 거부(치트).
→ "전투 무조건 승리" 같은 결과 위조 차단.
```
- **공유 모듈**(브라우저+Node 동일 코드)이 설계의 열쇠이자 **가장 큰 작업**. 풀로세움이 순수 JS라 가능한 이점.

### 결제(유료재화) — 서버 권위에 자연 포함
- 유료재화·유료 해금은 **Cloud Function만 write**, 그것도 **구글이 결제 검증(purchaseToken)** 후에만. 클라는 읽기만. → 공짜로 유료템 = 불가능.

## 3. 가장 큰 리스크/비용 (정직하게)

**전투·성장 로직을 `index.html`에서 떼어내 "결정론적 순수 공유 모듈"로 만드는 것.** 이 프로젝트에서 가장 무겁고 가장 중요한 단일 과제.
- 현재: 모든 게 단일 파일에 뒤엉킴 + `Math.random` 자유 사용 가정.
- 서버 재현을 위해 필요: ① RNG를 **시드 한 곳**으로(같은 시드=같은 전투) ② 시뮬 코어를 **DOM/렌더링과 분리된 순수 함수**로.
- 솔로로는 야심찬 규모. 단 launch가 멀어 시간 확보됨 + JS 공유로 난이도 완화.

## 4. 로드맵 재편 (마일스톤)

- **기둥 1 (서버 권위 계정)** 먼저 — 이것만으로 "치트 만렙" 불가. 크로스기기 세이브 포함. **출시는 이 위에서.**
- **기둥 2 (전투 재검증)** — PvP 전까지 완성(결정론 공유 시뮬).
- **결제** — 서버 권위 계정에 entitlements + 구글 결제검증 Cloud Function.
- **PvP/랭킹/시즌** — 기둥1·2 위 비동기(고스트) PvP. 랭크는 서버 권위 로스터로만.
- 백엔드: Firebase Auth + Firestore(계정 상태) + Cloud Functions(발급·검증·결제).

> GPT 원안 17단계 평가는 §7. 핵심 교정: "전체 Firestore 이전"(원안6)은 폐기가 아니라 **서버 권위 계정으로 격상**(blob 저장이 아니라 서버가 발급·보관), Cloud Function 검증 항목 명시, 치트 방지를 **출시 전제**로 끌어올림.

## 5. ⭐ 지금부터 지킬 규율 (비용 절감 — CLAUDE.md 하드 지침)

무거운 서버 작업은 나중이라도, **지금 추가하는 모든 전투/성장 코드**는 미래 추출이 싸지도록 아래를 지킨다. (구코드 일괄 개조 아님 — 신규/수정분에 규율 적용)

1. **전투 RNG 단일 시드화.** 시뮬 안에서 `Math.random()` 직접 호출 금지. 전투용 난수는 시드 기반 단일 함수 `rng()`(매치 시작 시 시드 주입) 하나로만. (배경 애니 등 *연출* 난수는 무관.)
2. **시뮬/렌더 분리.** 전투 결과 계산은 **순수**해야 한다 — resolve 경로에서 DOM 읽기·`Date.now()`·전역 가변 상태 의존 금지. 렌더/연출은 *결과를 읽기만* 하고 계산하지 않는다.
3. **신규 콘텐츠도 순수하게.** 새 스킬/종/효과의 수치는 `(상태, rng) → 결과` 순수 함수로 표현. 사이드채널(렌더 도중 상태 변경 등) 금지.
4. **시뮬 코어 결합 최소화.** 전투 로직을 UI 전용 전역에 묶지 않는다(나중에 모듈로 떼어낼 수 있게).

> 목적: 이 규율을 지키면 기둥2(서버 재검증)의 "결정론 공유 모듈 추출" 비용이 몇 배 싸진다.

## 6. 로그인 / 온라인 모델

- **익명 기본 + 구글 연동.** 시작은 익명 계정 자동, "구글로 백업/기기이전" 시 `linkWithCredential` 승격.
- **안드로이드:** WebView에서 `signInWithPopup` 안 됨 → **네이티브 플러그인 `@capacitor-firebase/authentication`** 로 네이티브 로그인 후 `signInWithCredential` 브리지. Firebase에 **SHA-1 등록** 필요. 웹은 팝업.
- **온라인 필수:** 진행이 서버를 거치므로 사실상 온라인 필수. (싱글 "연습" 모드를 보상 없이 오프라인 허용할지는 후속 선택.)
- **개인정보 처리방침:** 계정(PII) 저장 → Play Console 데이터보안 신고 + 처리방침 URL = **출시 필수**.
- **`?dex=1` 도감 모드:** 서버/Auth 미초기화(기존 부팅·세이브 스킵 분기 재사용).
