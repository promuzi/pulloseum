# 도트(픽셀아트) UI 전환 — 작업 인계 문서

> **다른 기기/새 세션의 Claude에게:** 사용자가 게임 UI를 도트(픽셀아트)로 바꾸려고 한다.
> 이 문서는 이전 대화에서 합의한 방향·결정·작업 흐름을 담은 인계서다. 도트/UI 적용 작업을 시작하기 전에 이 문서를 먼저 읽고 이어서 진행할 것.
> (작성일: 2026-06-20)

## 0. 한 줄 요약
게임의 식물·아이콘·UI를 **외부에서 만든 PNG 도트 그림으로 교체**한다. 캔버스로 게임을 새로 그리는 게 아니라, **기존 HTML/CSS 구조 위에 그림만 갈아끼우는 방식(①번)**으로 간다. 사용자가 그림을 규격 맞춰 제작하고, AI가 게임에 연결·모션 재생·전체 룩을 담당한다.

## 1. 채택한 방식: ①번 "PNG를 DOM에 교체"
- **이 게임은 메뉴·턴제 위주**(상점/탭/목록/모달)라, 화면이 매 프레임 다시 그려지지 않음 → PNG를 `<img>`로 붙여도 성능 문제 없음.
- 풀 캔버스 픽셀 렌더링(②번, 저해상도 캔버스 통째 확대)은 **액션게임용**이고, 이 장르엔 UI를 전부 다시 짜야 해서 오히려 비효율 → **채택 안 함.**
- 단, 함선 화면(`drawShip`/`SHIP_MAP`)은 이미 캔버스 픽셀 방식으로 되어 있음(참고).

### 핵심: 코드에 이미 교체 장치가 있다
- `index.html`의 `spriteFor(obj, size)` (대략 6626행 부근)이 식물 그림을 그림.
  ```js
  const key = spriteKey(seedType, growth, element);
  if(SPRITE_OVERRIDES[key]){
    return `<img src="${SPRITE_OVERRIDES[key]}" .../>`;  // 외부 PNG 있으면 그걸 씀
  }
  return composePlantSvg(...);  // 없으면 코드로 그린 SVG
  ```
- 즉 **`SPRITE_OVERRIDES` 표에 `key → PNG경로`만 추가하면** 식물 그림이 그 PNG로 바뀐다. 식물마다 코드 수정 불필요.
- `key`는 `spriteKey(seedType, growth, element)` (종자타입·생장단계·속성 조합). `NAME_OVERRIDES`도 같은 key로 이름 교체 가능.
- 프로필/배너는 `PROFILE_ART`/`BANNER_ART`(코드 생성 도트) — 같은 식으로 override 추가 가능.
- **결론: 그림 교체는 "그림 만들기 + 경로 연결"이고, 연결 작업은 AI가 한다.** 사용자가 일일이 코드 추가할 필요 없음.

## 2. 픽셀 크기/규격 개념 (사용자에게 설명 완료한 내용)
- **도트 한 칸 크기는 같은 카테고리 안에서 통일**해야 함: "식물끼리 같은 격자", "아이콘끼리 같은 격자". 식물 32도트 / 아이콘 16도트처럼 카테고리별로 다른 건 OK.
- **확대는 정수배만**(×2, ×3, ×4). ×1.5 등은 흐려짐. 코드엔 `image-rendering:pixelated` 이미 사용 중(함선 화면) — 도트 표시 필수 옵션.
- **식물은 모두 같은 캔버스 크기**로 그려야 정렬이 안 틀어짐. (`spriteFor`가 `height = size*140/120` 가정 → 현재 폭:높이 ≈ 120:140 ≈ 1:1.17)
- 격자 수 트레이드오프: 적을수록(예 16) 투박한 레트로 / 많을수록(예 64+) 디테일↑·도트맛↓·그릴 노동↑.

### 미확정 결정사항 (사용자가 정해야 함)
- [ ] **식물 격자 해상도** — 비교 도구로 12/16/24/32/48 보여줬고, 사용자는 "더 잘게(디테일)" 선호. 참고로 사용자가 보낸 샘플(노란 꽃 화분 GPT 그림)은 대략 폭 64~96도트급의 정교한 스타일(스타듀밸리 톤).
- [ ] **공통 캔버스 크기** — 샘플이 세로로 길었음(약 1:1.8). 모든 식물 들어갈 고정 칸 크기 + 식물/화분 위치 기준을 정해야 함.
- [ ] **아이콘 격자** — 보통 16~24. 현재 아이콘은 이모지 글자라, PNG로 바꾸려면 해당 자리 찾아 `<img>`로 교체(AI 작업).
- [ ] **어디까지 도트화** — 식물 → 아이콘 → 창틀(모달 테두리) 순서 추천. 창틀은 9-슬라이스(`border-image`)로 크기 가변 대응 필요(유일하게 손 가는 부분).

## 3. 그림 소스: AI 도트 생성 도구
사용자가 그림을 직접 안 그리고 AI로 뽑을 계획. 핵심 결론:
- **GPT 등 범용 이미지 AI = "픽셀처럼 보이는 그림"일 뿐 진짜 격자 도트가 아님.** 도트 칸 크기가 들쭉날쭉, 가장자리 흐릿, 생성마다 스타일 달라짐 → 여러 장 통일이 안 됨.
- **변환 파이프라인으로 갈 거면 GPT한테 "도트로 그려줘"는 비추** (가짜 픽셀 → 재축소 = 이중 픽셀화로 지저분). 대신 **"깔끔하고 단순한 플랫 일러스트, 또렷한 외곽선, 단순한 색, 투명/단색 배경"**으로 시키고, 도트 변환은 마지막에 통일해서 입히는 게 나음.

### 추천 도트 전용 AI (둘 다 유료, 무료 체험 크레딧 있음 — 가격은 사이트에서 직접 확인)
| 도구 | 강점 | 추천 상황 |
|---|---|---|
| **PixelLab.ai** | 게임 캐릭터+**애니메이션 전용**. 스켈레톤 기반으로 모션 프레임을 일관되게 생성. img2img(참고 그림 입력) O. 4방향·타일셋. Aseprite 플러그인+웹 | **모션까지 갈 거면 1순위** |
| **Retro Diffusion** | 정적 도트 퀄리티·팔레트/스타일 통일 강함. img2img O. Aseprite 플러그인 우수 | 정적 스타일 통일 중시 |

- **낙서(참고 그림) 입력**: 둘 다 가능(image-to-image). 낙서로 형태·포즈·구도 잡아주면 따라 그림. 낙서 + 짧은 텍스트("노란 꽃, 화분, 정면, 웃는 얼굴") 병행 시 적중률↑.
- **사용자에게 권한 다음 행동**: PixelLab 무료 크레딧으로 캐릭터 1종 만들고 → 낙서로 형태 잡고 → idle 모션 몇 프레임 뽑아 일관성 확인.
- 보조 도트 툴: Aseprite(유료, 후보정 최고), Piskel/LibreSprite(무료).

## 4. 모션(애니메이션) 계획 — ①번에서 가능함
- 전투에 식물 2마리뿐 → 성능 걱정 없음.
- **두 종류 모션:**
  1. **통짜 모션(transform)** — 둥실/흔들/돌진/쓰러짐. **이미 코드에 있음**: `shake`, `dmgpop`, `collapse`, `fighterIn` 등 CSS 애니메이션. PNG로 바꿔도 그대로 작동.
  2. **프레임 모션(스프라이트시트)** — 잎 살랑·스킬 불꽃처럼 그림 자체가 바뀌는 진짜 도트 애니. **여러 프레임을 PNG 한 장에** 가로로 늘어놓고 CSS `steps()` 애니메이션으로 칸 넘김.
- **규격**: 프레임 2~6개면 충분(idle 4, 공격 6 등). **모든 프레임 같은 캔버스 크기**(예 칸 120×140 × 4프레임 = 480×140). 스킬 이펙트는 식물과 **별개 PNG 레이어**로(재활용).
- 일관된 모션의 핵심: **캐릭터 1종을 먼저 확정 → 그 캐릭터로 모션만 생성**(PixelLab이 이 구조라 프레임 간 일관성 유지). 매번 새로 뽑으면 캐릭터가 달라짐.

## 5. AI 측 변환 보조 (이 PC 환경)
- 이 PC: **Node.js v24 있음. Python PIL / ImageMagick 없음.**
- 변환 원리: **고정 격자로 다운스케일 + 팔레트(색 수) 축소** → 균일한 진짜 도트.
- 채팅에 붙여넣은 그림은 AI가 파일로 못 읽음 → **사용자가 PNG를 프로젝트 폴더(예 `assets/raw/`)에 저장**해야 변환 스크립트 실행 가능.
- 필요 시 Node 스크립트(jimp 등 순수 JS 라이브러리)로 일괄 변환 가능. (단 이미 디테일 많은 그림은 축소 시 일부 뭉개짐 → 단순·플랫한 소스일수록 변환 잘 됨.)

## 6. 작업 흐름 (역할 분담)
```
1. 사용자: 식물 격자 크기 + 공통 캔버스 크기 결정 (2장의 "미확정 결정사항")
2. 사용자: AI 도트툴로 그림 1종 제작 (규격대로, 낙서 참고 가능)
3. AI: assets에 저장 + SPRITE_OVERRIDES에 연결 → preview로 확인
4. 같이: 크기·정렬 OK? → 규격 확정 (★여기서 캔버스 크기/여백 못 박기)
5. 사용자: 나머지 그림 같은 규격으로 제작
6. AI: 들어오는 대로 연결 + 모션 재생(steps 애니메이션) 반복
```
- **개별 그림 = 사용자 제작 / 연결·모션·전체룩 = AI.**
- **전체 분위기(픽셀 폰트·레트로 색·각진 테두리)**는 CSS로 한 번에 적용 가능(그림과 별개로 먼저 해볼 수도 있음).

## 7. 다음 AI가 시작할 때 추천 첫 행동
1. 사용자에게 "식물 격자 크기 / 공통 캔버스 크기" 정했는지 물어보기 (2장 체크리스트).
2. 미정이면 비교용으로 도와주고, 정해지면 **올바른 크기의 투명 PNG 템플릿 + 가이드**를 만들어 주면 규격 통일이 쉬움.
3. 사용자가 그림 파일을 `assets/`에 넣으면 `SPRITE_OVERRIDES`에 연결 → preview 검증(이 환경은 preview 스크린샷이 타임아웃날 수 있어 DOM/eval로 검증).
4. 검증·구조 주의사항은 루트 `CLAUDE.md` 참고(게임 전체가 `index.html` 단일 파일, 데이터 인라인).

## 8. 역할 분담 확정 (2026-06-21 보강)
사용자가 원하는 건 **"도트 흉내(가짜 픽셀) vs 진짜 도트 형식"의 구분**이었고, 디테일·의도가 담긴 그림을 직접 디렉팅하길 원함. 그래서 분담을 이렇게 확정:
- **도트 AI(사용자가 디렉팅)** — 디테일·의도 중요한 "보여주는 그림": 식물 스프라이트(생장단계·속성별), 주요 배경, 큰 장식, 모션 프레임. → 진짜 도트 형식 + 일관되게.
- **AI(Claude)** — UI 골격·기능: 창틀/모달/패널, 버튼, HUD, 체력·에너지 바, 레이아웃, 픽셀 폰트, 색 테마, **AI 그림을 게임에 연결(SPRITE_OVERRIDES)**, 모션 재생(CSS steps), 정수배 확대, 전체 톤 통일.
- **애매지대(아이콘)**: "예쁘게 보여주는 것"이면 AI 그림 / "기능 표시"면 Claude. 개별 결정.
- ⚠️ **Claude는 디테일 식물·배경을 직접 그리지 않음**(사용자가 못생길까 우려한 부분). Claude 역할은 구조·스타일·연결.
- `pixelSvg`(도트 데이터 직접 렌더) 방식은 단순 도트엔 가볍고 좋지만, 디테일·의도 중시하는 이 방향엔 안 맞아 **보류**(선택지로만).

### 분담 성립 조건 — 공통 규격 3개 (가장 먼저 합의할 것)
AI 그림과 Claude UI가 같은 게임으로 보이려면:
1. **색 팔레트** 통일
2. **도트 한 칸 겉보기 크기** 통일(그림 도트 ≈ UI 테두리·폰트 도트)
3. **픽셀 폰트** 톤 통일
→ 식물 AI로 1종 먼저 만들면서 이 3개를 같이 잡는 게 자연스러움.

## 10. UI 미세 수정을 사용자가 효과적으로 전달하는 법
사용자는 코딩으로 버튼 위치·크기를 직접 못 고침. 사진만으론 정확도 떨어짐. 효과적 순서:
1. **번호 라벨(★최우선)** — Claude가 preview에서 각 요소 위에 ①②③ 임시 번호를 띄움 → 사용자는 "3번을 더 크게, 5번을 2번 왼쪽으로"처럼 번호로 지칭. 혼동 제거.
2. **짧은 핑퐁 루프** — 한 방에 설명 말고, Claude가 바꿔 보여주면 "더 왼쪽/너무 큼/딱 좋아"로 반응. 2~3회면 픽셀 단위까지 맞음.
3. **요청 공식** — [무엇(번호/이름)] + [어떻게] + [기준/비교]. 예 "③을 ⑤랑 같은 크기로 화면 가운데".
4. **스크린샷 마킹** — 사진 위에 동그라미(대상)+화살표(방향)+짧은 글씨로 표시.
5. **(선택) 조절판 위젯** — Claude가 크기·위치 슬라이더 위젯 제공 → 사용자가 눈으로 맞춘 값만 전달.
- 더 발전된 옵션(미도입): 브라우저 DevTools 라이브 편집, 게임 내 "디자인 모드"(드래그·리사이즈 핸들→CSS값 출력), Figma/Canva 등으로 정밀 목업. (실험 가치 있음.)
